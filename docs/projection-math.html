The Depth Projection Math Behind Planespace

Published 2024 · 12 min read · Technical

Planespace makes HTML elements appear to float at different depths in 3D space without a scene graph, WebGL, or any modification to the DOM's layout. To understand how this works, you need to understand three things: how we map Z-values to a depth texture, how viewer position maps to a parallax offset, and how a single fullscreen shader composites everything into the final frame.

This post covers the projection math precisely, including the derivations that informed our choices.

Step 1: From data-z to Normalized Depth

Every element with a data-z attribute declares a position in Z-space. We first convert these raw user values into a normalized [0, 1] range that the shader can consume:

normalizedDepth(z) = clamp( (z − depthRange[0]) / (depthRange[1] − depthRange[0]), 0, 1 )

With the default depthRange: [-600, 100], an element at z = -600 gets depth 0 (far background), and an element at z = 100 gets depth 1 (near foreground). This linearity is a deliberate choice: it keeps the math predictable and the scene editable without surprises.

The depth projection is deterministic and documented in SPEC.md as Invariant I-6. Any given set of elements and depthRange always produces the same depth map.

Step 2: Building the Depth Texture

Once we have normalized depth values per element, we need a per-pixel representation. The DepthRegistry.generateDepthTexture(width, height) method rasterizes element bounding boxes into a Float32Array:

for (const { el, z } of entries) {
  const rect = el.getBoundingClientRect();
  const depth = normalize(z);

  // Convert viewport rect to pixel coords
  const x0 = Math.floor(rect.left   * (width  / window.innerWidth));
  const y0 = Math.floor(rect.top    * (height / window.innerHeight));
  const x1 = Math.ceil (rect.right  * (width  / window.innerWidth));
  const y1 = Math.ceil (rect.bottom * (height / window.innerHeight));

  for (let py = y0; py < y1; py++) {
    for (let px = x0; px < x1; px++) {
      const idx = py * width + px;
      // Nearest-to-viewer wins
      if (depth > data[idx]) data[idx] = depth;
    }
  }
}

The result is a grayscale float texture where each pixel encodes how "near" the frontmost depth element is at that screen location. Pixels with no element default to 0 (background depth).

Step 3: Viewer Angle to Parallax Offset

The mouse position in the viewport maps to a normalized viewer angle. With the cursor at the horizontal center, rx = 0. Far right gives rx = 1, far left gives rx = -1:

rx = (mouseX / viewportWidth) × 2 − 1 ry = (mouseY / viewportHeight) × 2 − 1

This raw value is smoothed via exponential moving average before use:

viewerX += (targetX − viewerX) × lerpFactor

With lerpFactor = 0.06, the smoothed angle reaches 95% of the target after roughly 50 frames (~0.83 seconds at 60fps). This creates the characteristic "floating" feel.

Step 4: The Warp Shader

In reproject mode, the scene is captured as an image and passed to a WebGL2 fragment shader alongside the depth texture. The shader runs once per pixel of the output canvas:

#version 300 es
precision highp float;

uniform sampler2D T_scene;   // captured DOM frame
uniform sampler2D T_depth;   // normalized depth map
uniform vec2 viewerAngle;    // smoothed viewer rx, ry
uniform float warpStrength;  // parallax intensity

in vec2 vUV;
out vec4 fragColor;

void main() {
  // 1. Sample depth at this pixel
  float depth = texture(T_depth, vUV).r;

  // 2. Compute UV offset: near pixels shift more
  vec2 offset = viewerAngle * depth * warpStrength;

  // 3. Sample scene at warped UV
  vec2 sampleUV = clamp(vUV + offset, 0.001, 0.999);
  fragColor = texture(T_scene, sampleUV);
}

The key line is offset = viewerAngle * depth * warpStrength. This is a direct implementation of the parallax formula: the shift of a pixel is proportional to its apparent depth. Near objects (depth ≈ 1) shift by viewerAngle × warpStrength, while background pixels (depth ≈ 0) don't shift at all.

Why UV space?

The offset is computed in UV space (0 to 1) rather than pixel space. This makes it viewport-size-independent — the same warpStrength value produces the same perceptual parallax intensity regardless of whether you're on a 1080p monitor or a 4K display.

Edge clamping

Without clamping, warped UVs outside [0, 1] would wrap or mirror, creating jarring edge artifacts. We clamp to a tight inset of 0.001 rather than exactly 0 to avoid any sub-pixel bleed from the texture border.

Step 5: Transform Mode Projection

When WebGL2 is unavailable, Planespace falls back to CSS preserve-3d. The math here is classical perspective projection, offloaded entirely to the browser's compositor:

container.style.transform = `perspective(${P}px) rotateY(${rx × maxAngle}deg) rotateX(${−ry × maxAngle}deg)`

Each depth element receives translateZ(z)px. The browser projects these into screen space using standard perspective division:

screenX = (worldX × P) / (P − z) screenY = (worldY × P) / (P − z)

Where P is the perspective distance in pixels (default 900). Near elements (z positive) appear larger and shift more during rotation. This is exactly the correct perceptual behavior.

Temporal Smoothing

The depth texture is expensive to regenerate (it requires a pass over all element bounding boxes). We avoid regenerating it every frame unless the DOM changes. Between regenerations, we apply temporal smoothing to prevent visible flickering on frame boundaries:

finalDepth[i] = prevDepth[i] × temporalSmoothing + currentDepth[i] × (1 − temporalSmoothing)

With temporalSmoothing = 0.85, this is an 85/15 blend of the previous and current depth map. It acts as a low-pass filter on the depth texture, smoothing out any per-frame jitter from getBoundingClientRect() precision.

Chromatic Aberration (Optional)

Near-plane elements can optionally render with RGB channel separation — simulating the lens aberration of a real camera pointed at a near object. The effect scales with depth:

chromaticOffset = depth × chromaticStrength R = sample(T_scene, sampleUV + vec2(ca, 0.0)).r G = sample(T_scene, sampleUV).g B = sample(T_scene, sampleUV − vec2(ca, 0.0)).b

Background pixels (depth ≈ 0) see zero channel split; foreground pixels get the full offset. This creates a subtle visual cue that near objects are physically "in front" of the focal plane.

What This Isn't

This is not ray casting, not volume rendering, and not a proper 3D projection. We are remapping UV coordinates based on a simplified depth signal derived from DOM bounding boxes. Overlapping elements at different depths will have a rectangular depth boundary at their box edges — not a per-pixel alpha depth edge. This is an intentional tradeoff: it keeps the system DOM-native, deterministic, and O(n) in element count.

For scenes with complex transparency and non-rectangular depth boundaries, a proper 3D engine is the right tool. Planespace is for the 95% of web UIs that are made of rectangular boxes and benefit from perceptual depth without the overhead of a retained scene graph.
